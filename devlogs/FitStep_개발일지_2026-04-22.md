## 1. 운동 GIF 이미지 미표시 — RapidAPI ExerciseDB `gifUrl` 필드 제거

### 문제
AI 운동 루틴 추천 페이지에서 운동 GIF 이미지가 전혀 표시되지 않음.
MySQL Workbench로 확인 시 `exercises` 테이블의 `gif_url` 컬럼이 모두 `NULL`.

### 원인 분석
기존 코드에서 RapidAPI ExerciseDB의 `/exercises/name/{name}` 엔드포인트를 호출하고
응답의 `gifUrl` 필드를 읽어 DB에 저장하는 방식이었음.

```python
# 기존 코드 (동작 안 함)
data = response.json()
gif_url = data[0].get("gifUrl")  # → 항상 None
```

API 응답을 직접 확인한 결과, RapidAPI ExerciseDB가 스키마를 변경하여
**`gifUrl` 필드를 응답에서 완전히 제거**한 것이 원인으로 확인됨.

변경 전후 비교:
| 필드 | 변경 전 | 변경 후 |
|------|--------|--------|
| `gifUrl` | GIF 이미지 URL 포함 | **필드 자체 없음** |
| `exerciseId` | 포함 | 포함 (유지) |

### 해결 방법
새로운 RapidAPI ExerciseDB `/image` 엔드포인트 발견:
- `GET /image?exerciseId={id}&resolution={360|480|720}` → 실제 GIF 바이너리 반환

**2단계 프록시 아키텍처로 전환:**

1. 인증된 엔드포인트(`/db/exercises/gif`)에서 운동명으로 `exerciseId` 조회 후 DB 캐싱
2. 인증 없는 프록시 엔드포인트(`/db/exercises/gif-proxy`)에서 GIF 바이너리를 StreamingResponse로 반환

### 구현 내용

#### `08_FitStep_API/app/db_router.py` — gif 조회 및 프록시 엔드포인트 추가

```python
@router.get("/exercises/gif")
async def get_exercise_gif(name_kr: str = Query(...), name_en: str = Query(...)):
    # 1) DB 캐시 확인
    conn = get_connection()
    cursor = conn.cursor(dictionary=True)
    cursor.execute("SELECT gif_url, exercise_id FROM exercises WHERE name = %s OR name_en = %s", (name_kr, name_en))
    row = cursor.fetchone()

    # 이미 exercise_id가 캐시돼 있으면 프록시 URL 바로 반환
    if row and row.get("exercise_id"):
        exercise_id = row["exercise_id"]
        gif_url = f"/db/exercises/gif-proxy?exerciseId={exercise_id}"
        return {"gif_url": gif_url}

    # 2) RapidAPI에서 exerciseId 검색
    async with httpx.AsyncClient(timeout=15.0) as client:
        resp = await client.get(
            f"https://exercisedb.p.rapidapi.com/exercises/name/{name_en}",
            headers={"X-RapidAPI-Key": RAPIDAPI_KEY, "X-RapidAPI-Host": "exercisedb.p.rapidapi.com"}
        )
    data = resp.json()
    if not data:
        return {"gif_url": None}

    exercise_id = data[0].get("exerciseId") or data[0].get("id")

    # 3) DB에 exercise_id 캐싱
    cursor.execute("UPDATE exercises SET exercise_id = %s WHERE name = %s OR name_en = %s",
                   (exercise_id, name_kr, name_en))
    conn.commit()

    gif_url = f"/db/exercises/gif-proxy?exerciseId={exercise_id}"
    return {"gif_url": gif_url}


# 인증 없는 별도 라우터 — Streamlit st.image()가 직접 접근 가능해야 하므로
gif_proxy_router = APIRouter()

@gif_proxy_router.get("/gif-proxy")
async def gif_proxy(exerciseId: str = Query(...)):
    async with httpx.AsyncClient(timeout=15.0) as client:
        resp = await client.get(
            f"https://exercisedb.p.rapidapi.com/image?exerciseId={exerciseId}&resolution=360",
            headers={"X-RapidAPI-Key": RAPIDAPI_KEY, "X-RapidAPI-Host": "exercisedb.p.rapidapi.com"}
        )
    return StreamingResponse(
        iter([resp.content]),
        media_type=resp.headers.get("content-type", "image/gif")
    )
```

#### `07_FitStep_Web/api_client.py` — 상대경로 → 절대경로 변환

```python
def api_get_exercise_gif(name_kr, name_en):
    try:
        res = _get("/db/exercises/gif", name_kr=name_kr, name_en=name_en)
        gif_url = res.get("gif_url")
        # 프록시 URL이 상대경로이므로 베이스 URL 붙여서 절대경로로 변환
        if gif_url and gif_url.startswith("/"):
            gif_url = _base() + gif_url
        return gif_url
    except Exception as e:
        print(f"Error fetching gif for {name_kr}: {e}")
        return None
```

### 핵심 트러블슈팅 포인트

**① 프록시 엔드포인트 인증 문제**

처음에 `gif-proxy`를 기존 `router`(= `APIRouter(dependencies=[Depends(verify_api_key)])`) 안에 만들었더니,
Streamlit의 `st.image(url)`이 URL을 직접 요청할 때 API 키 헤더를 추가할 방법이 없어 401 오류 발생.

→ `gif_proxy_router = APIRouter()` 를 별도로 만들어 인증 의존성 없이 `main.py`에 독립 등록.

**② 상대경로 vs 절대경로**

FastAPI가 반환한 프록시 URL이 `/db/exercises/gif-proxy?exerciseId=0025` 형식의 상대경로.
`st.image()`에는 절대 URL이 필요하므로, `api_client.py`에서 `_base()` (= `RAG_API_URL`)를 앞에 붙여 변환.

**③ RapidAPI 신규 엔드포인트 탐색 방법**

`/image` 엔드포인트를 파라미터 없이 호출했을 때 API가 친절하게 필수 파라미터 목록(`exerciseId`, `resolution`)을 오류 메시지로 알려줌.
→ `exerciseId=0025&resolution=360`으로 테스트하니 124KB GIF 바이너리 정상 반환 확인.

### 결과
- 운동 루틴 추천 페이지에서 각 운동의 GIF 애니메이션 정상 표시
- 첫 조회 시 RapidAPI 호출 → `exercise_id` DB 캐싱 → 이후 요청은 캐시 활용으로 API 호출 횟수 절감

## 2. GIF 이미지 로딩 전체 흐름 상세

### URL 검색 및 GIF 로딩 흐름

```
Streamlit (브라우저)
    │
    │ api_get_exercise_gif(name_kr="바벨 벤치프레스", name_en="barbell bench press")
    ▼
api_client.py
    │ GET /db/exercises/gif?name_kr=...&name_en=...
    ▼
FastAPI /db/exercises/gif  (인증 필요)
    │
    ├─ [1단계] MySQL 조회: SELECT exercise_id FROM exercises WHERE name = name_kr
    │
    ├─ exercise_id가 이미 있으면 → gif_url = "/db/exercises/gif-proxy?exerciseId=0025" 반환 (끝)
    │
    └─ exercise_id가 없으면 →
         │
         │ [2단계] RapidAPI 이름 검색
         │ GET https://exercisedb.p.rapidapi.com/exercises/name/barbell%20bench%20press?limit=1
         │ 응답: [{"id": "0025", "name": "barbell bench press", "bodyPart": "chest", ...}]
         │
         ├─ [3단계] exercise_id DB 저장
         │ UPDATE exercises SET exercise_id = '0025' WHERE name = '바벨 벤치프레스'
         │
         └─ gif_url = "/db/exercises/gif-proxy?exerciseId=0025" 반환

api_client.py
    │ 상대경로 → 절대경로 변환
    │ "http://서버주소/db/exercises/gif-proxy?exerciseId=0025"
    ▼
Streamlit
    │ st.image("http://서버주소/db/exercises/gif-proxy?exerciseId=0025")
    │ 브라우저가 직접 URL 요청
    ▼
FastAPI /db/exercises/gif-proxy  (인증 없음)
    │ GET https://exercisedb.p.rapidapi.com/image?exerciseId=0025&resolution=360
    │ → GIF 바이너리(124KB) 수신
    ▼
StreamingResponse(iter([resp.content]), media_type="image/gif")
    ▼
브라우저에 GIF 표시
```

### DB 저장 구조

`exercises` 테이블이 GIF 관련 캐시 역할을 담당:

| 컬럼 | 타입 | 역할 |
|------|------|------|
| `name` | VARCHAR(100) | 한글 운동명 (GPT가 생성하는 이름과 매칭 키) |
| `name_en` | VARCHAR(150) | 영문 운동명 (RapidAPI 검색에 사용) |
| `exercise_id` | VARCHAR(20) | RapidAPI의 운동 고유 ID (GIF 프록시 URL에 사용) |
| `gif_url` | VARCHAR(512) | 구 방식 잔재 (현재 NULL, 미사용) |
| `synced` | TINYINT | RapidAPI 전체 동기화 여부 (1=완료) |
| `body_part` | VARCHAR(50) | 운동 부위 |

**데이터 저장 시점:**

1. **루틴 저장 시** (`POST /db/routines`): GPT가 생성한 운동 목록을 `exercises` 테이블에 INSERT  
   → 이 시점엔 `name`, `name_en`, `category`만 저장. `exercise_id`는 NULL.

2. **GIF 조회 시** (`GET /db/exercises/gif`): 처음 이미지 요청할 때 RapidAPI 검색 후 `exercise_id` UPDATE  
   → 이후 같은 운동 요청은 DB 캐시만 사용, RapidAPI name 검색 생략.

**실제 저장 예시:**

```
루틴 저장 직후:
┌────┬──────────────────┬─────────────────────┬─────────────┬────────────┐
│ id │ name             │ name_en             │ exercise_id │ synced     │
├────┼──────────────────┼─────────────────────┼─────────────┼────────────┤
│  1 │ 바벨 벤치프레스  │ barbell bench press │ NULL        │ 0          │
│  2 │ 스쿼트           │ squat               │ NULL        │ 0          │
└────┴──────────────────┴─────────────────────┴─────────────┴────────────┘

첫 GIF 조회 후 (캐싱 완료):
┌────┬──────────────────┬─────────────────────┬─────────────┬────────────┐
│ id │ name             │ name_en             │ exercise_id │ synced     │
├────┼──────────────────┼─────────────────────┼─────────────┼────────────┤
│  1 │ 바벨 벤치프레스  │ barbell bench press │ 0025        │ 0          │
│  2 │ 스쿼트           │ squat               │ 0190        │ 0          │
└────┴──────────────────┴─────────────────────┴─────────────┴────────────┘
```

### RapidAPI 엔드포인트 정리

| 엔드포인트 | 용도 | 반환 |
|-----------|------|------|
| `GET /exercises/name/{name}?limit=1` | 영문 운동명으로 `exerciseId` 검색 | JSON 배열 (`id`, `name`, `bodyPart` 등) |
| `GET /image?exerciseId={id}&resolution=360` | exerciseId로 GIF 바이너리 직접 수신 | 이미지 바이너리 (image/gif) |
| `GET /exercises?limit=1500` | 전체 운동 목록 동기화 (sync 엔드포인트용) | JSON 배열 |
