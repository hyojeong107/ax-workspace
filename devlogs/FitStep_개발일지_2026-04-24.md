### 1. RAG 프레임워크 LangChain으로 전환

**배경**
기존 RAG는 ChromaDB SDK + OpenAI SDK를 직접 호출하는 방식이었음. LangChain 체인 기능(Agent, Retriever 등) 연동을 위해 전환 결정.

**변경 파일**
- `08_FitStep_API/app/indexing.py`
  - `chromadb.PersistentClient` → `langchain_chroma.Chroma`
  - `OpenAI().embeddings.create()` → `langchain_openai.OpenAIEmbeddings`
  - `chromadb.Collection.add()` → `Chroma.add_documents(documents=[Document(...)])`
  - `_get_collection()` → `_get_gym_store()`로 rename

- `08_FitStep_API/app/retrieval.py`
  - `_get_collection()` import → `_get_gym_store()` import로 변경
  - 로직은 동일하게 유지 (get/delete는 LangChain Chroma가 ChromaDB API 그대로 노출)

- `08_FitStep_API/requirements.txt`
  - 추가: `langchain>=0.3.0`, `langchain-openai>=0.2.0`, `langchain-chroma>=0.1.0`

**설치**
```bash
venv\Scripts\pip install langchain langchain-openai langchain-chroma
```

### 3. 공공데이터 초기 인덱싱 스크립트 작성 (Phase 0 - 인덱싱 파이프라인 고도화)

**파일**: `08_FitStep_API/init_public_data.py`

**적용된 고도화 전략 4가지**

#### 전략 1. 문서 종류별 인덱싱 전략 분리
- 체력측정: 1레코드 → 자연어 문장 1개로 변환 후 1문서로 저장
  - 예) `"64세 여성, 신장 158.7cm, 체중 51.7kg, BMI 24.4, 윗몸일으키기 22회. 등급: 4등급."`
- 운동추천: (연령대+BMI등급+성별+상장등급+운동단계) 조합 → 추천운동 묶어서 1문서
  - 예) `"30대 여성 1단계비만 1등급 준비운동: 걷기, 스트레칭, 자전거타기"`
- 헬스장 기구: 기존 방식 유지 (요약 1개 + 기구별 개별)

#### 전략 2. 청킹 전 문서 구조 보존 전처리
- `MESURE_IEM_XXX_VALUE` 코드 → 한글 필드명 변환
  - 001=신장, 002=체중, 003=체지방률, 005=허리둘레, 006=수축기혈압, 007=BMI, 008=윗몸일으키기, 018=제자리멀리뛰기
- null 값 필드 문서에서 제외 (빈 정보로 임베딩 품질 저하 방지)
- 성별코드 변환: M→남성, F→여성
- CSV 따옴표/공백 strip 처리

#### 전략 3. 검색에 유리한 메타데이터 확장
- 체력측정 메타데이터: `source`, `age_group`, `gender`, `grade`, `bmi`(수치), `bmi_category`(저체중/정상/과체중/비만), `age`(정수)
- 운동추천 메타데이터: `source`, `age_group`, `gender`, `bmi_grade`, `award_grade`, `exercise_step`, `exercise_count`

#### 전략 4. 증분 인덱싱 지원
- `force_reindex=False` (기본): 컬렉션에 데이터 있으면 스킵
- `force_reindex=True`: 기존 삭제 후 재인덱싱
- 실행: `python init_public_data.py` / `python init_public_data.py --force`

**ChromaDB 컬렉션 구조 (최종)**
```
chroma_db/
├── gym_equipment          # 사용자별 헬스장 기구 (user_id 필터)
├── fitness_measurement    # 체력측정 공공데이터 950건
└── exercise_recommendation # 운동추천 공공데이터 (조합별 묶음)
```

### 5. Phase 2 - POST /chat 엔드포인트 추가

**파일**: `app/schemas.py`, `app/main.py`

**문제점 (개선 전)**: retrieval 함수만 있고 실제로 AI 답변을 생성하는 흐름이 없었음 → RAG가 실질적으로 동작하지 않는 구조

**개선**: `POST /chat` 엔드포인트 신설
- 요청: `user_id`, `message`, `age`, `gender`, `bmi`, `age_group`, `bmi_grade`
- 흐름: 헬스장 컨텍스트 + 체력측정 유사사례 + 운동추천 3개 RAG 수집 → 시스템 프롬프트 주입 → GPT-4o-mini 호출 → 답변 반환
- 컨텍스트 없으면 일반 운동 조언으로 graceful fallback
