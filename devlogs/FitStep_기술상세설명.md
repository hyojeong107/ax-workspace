## 1. 전체 데이터 흐름 요약

사용자가 "루틴 추천" 버튼을 누르면 다음 순서로 동작합니다.

```
① 사용자 컨디션 입력 (슬라이더 + 근육통 부위 선택)
        ↓
② 최근 운동 이력 조회 (MySQL workout_logs)
        ↓
③ 쿨다운 계산 — 오늘 쉬어야 할 부위 결정 (Python)
        ↓
④ 헬스장 기구 정보 검색 (ChromaDB RAG)
        ↓
⑤ 공공데이터 유사 사례 검색 (ChromaDB RAG)
        ↓
⑥ 전체 컨텍스트를 GPT 프롬프트에 주입
        ↓
⑦ GPT → 루틴 JSON 생성
        ↓
⑧ MySQL routines 테이블에 저장
        ↓
⑨ 웹 화면에 카드·차트로 표시
```

## 2. 실제 사례로 보는 작동 방식

> **사용자 설정**: 김지수 / 35세 여성 / 165cm 62kg / BMI 22.8 (정상) / 체력 중급 / 목표: 체중감량+근육증가 / 부상: 무릎 통증 / 헬스장: 스미스 머신, 덤벨(5~30kg), 케이블 머신 보유

### 시나리오: 오늘 루틴 추천 버튼을 눌렀을 때

**Step 1 — 사용자가 컨디션 입력**

```
오늘 컨디션: 피곤 (2/5)
근육통 부위: 어깨 선택
```

**Step 2 — Python이 운동 이력 분석 (DB 조회)**

```
workout_logs 최근 2일 조회 결과:
  어제: 벤치프레스, 딥스, 오버헤드프레스 → 가슴, 삼두, 어깨 훈련
  이틀 전: 스쿼트, 런지 → 하체 훈련

쿨다운 계산:
  가슴 → 1일 경과, 기준 2일 → ❌ 쿨다운 중
  어깨 → 1일 경과, 기준 2일 → ❌ 쿨다운 중
  하체 → 2일 경과, 기준 3일 → ❌ 쿨다운 중
  등   → 4일 경과, 기준 3일 → ✅ 수행 가능 (가장 오래 쉼 → 우선 추천)
  복근 → 3일 경과, 기준 1일 → ✅ 수행 가능

점진적 과부하 계산:
  랫풀다운: 3회 수행, 최근 3세트×12회/40kg → ⬆ 레벨업 권장 → 43kg 제안
  플랭크: 2회 수행, 최근 3세트×45초 → → 현행 유지
```

**Step 3 — ChromaDB에서 헬스장 기구 조회 (필터 조회)**

```
user_id=5 필터로 ChromaDB 조회
→ "스미스 머신, 덤벨 5~30kg, 케이블 머신" 텍스트 반환
```

**Step 4 — ChromaDB에서 공공데이터 벡터 검색 (진짜 RAG)**

```
검색 쿼리: "35세 여성 BMI 22.8 정상"
→ 유사 체력측정 사례: "30대 여성 BMI 정상 체력등급 2등급, 근력 보통, 심폐 보통"
→ 운동추천: "30대 여성 정상체중: 스쿼트, 케이블 로우, 빠르게 걷기 권장"
```

**Step 5 — 프롬프트 조립 (RAG + Python 계산 + 사용자 입력 합체)**

```
[사용자 정보]        ← DB에서 조회
[부상 주의사항]      ← injury_tags "무릎" → Python 하드코딩 규칙
  무릎 통증: 풀스쿼트, 레그익스텐션 제외. 레그프레스(얕은 각도), 힙스러스트 권장.

[오늘 볼륨 조정]     ← 컨디션 "피곤" → Python 조건문
  운동 3-4개, 2-3세트, 강도 70-80%

[오늘 제외할 부위]   ← 최근 2일 이력 + 근육통 선택 → Python 계산
  가슴, 삼두, 어깨, 하체

[점진적 과부하]      ← workout_logs Python 계산
  랫풀다운: ⬆ 43kg으로 레벨업 권장

[쿨다운 스케줄]      ← 마지막 수행일 Python 계산
  오늘 추천: 등(4일), 복근(3일) / 제외: 가슴, 어깨, 하체

[헬스장 기구]        ← ChromaDB 필터 조회 (RAG ①)
  스미스 머신, 덤벨 5~30kg, 케이블 머신만 사용할 것

[공공데이터 사례]    ← ChromaDB 벡터 검색 (RAG ②)
  30대 여성 정상체중 권장: 케이블 로우, 빠르게 걷기
```

**Step 6 — GPT가 위 프롬프트를 받아 JSON 생성**

```json
{
  "exercises": [
    {"name": "케이블 랫풀다운", "name_en": "lat pulldown", "category": "등",
     "sets": 3, "reps": 12, "weight_kg": 43.0, "tip": "광배근 집중"},
    {"name": "케이블 시티드 로우", "name_en": "cable seated row", "category": "등",
     "sets": 3, "reps": 12, "weight_kg": 35.0, "tip": "허리 직립 유지"},
    {"name": "덤벨 루마니안 데드리프트", "name_en": "dumbbell romanian deadlift", "category": "등/햄스트링",
     "sets": 2, "reps": 12, "weight_kg": 15.0, "tip": "무릎 살짝 구부린 채 고관절 힌지"},
    {"name": "플랭크", "name_en": "plank", "category": "복근",
     "sets": 3, "reps": 0, "weight_kg": 0, "tip": "45초 유지"}
  ],
  "advice": "오늘은 피곤한 상태이므로 볼륨을 줄였습니다. 등과 복근 위주로 구성했으며, 무릎에 부담 없는 동작으로만 선택했습니다."
}
```

> GPT가 스쿼트나 레그프레스를 넣지 않은 이유: 프롬프트에 "하체 제외"와 "무릎 통증 주의사항"이 명시됐기 때문. GPT가 스스로 판단한 게 아니라 Python이 계산한 규칙을 따른 것.

**Step 7 — 화면 표시 시 GIF 조회 (루틴과 별개 동작)**

```
GPT가 반환한 "lat pulldown" → api_get_exercise_gif("케이블 랫풀다운", "lat pulldown") 호출
→ exercises 테이블에 gif_url 있으면 바로 표시
→ 없으면 RapidAPI에서 "lat pulldown" 검색 → GIF 다운로드 → 로컬 저장 → 표시
```

### 헬스장 환경 미등록 시

```
ChromaDB 조회 결과 없음
→ 프롬프트 [헬스장 기구] 섹션에 기본값 주입:
  "바벨, 덤벨, 케이블 머신, 스미스 머신, 레그프레스, 랫풀다운 등 모든 기구 자유롭게 사용 가능"
→ GPT가 일반 헬스장 기준으로 루틴 생성
```

## 3. RAG — 헬스장 기구 개인화

### 저장 단계 (사용자가 기구 등록할 때)

사용자가 본인 헬스장 기구를 입력하면, 해당 정보를 **텍스트로 변환 → 임베딩 → ChromaDB 저장**합니다.

```
입력 예시:
  헬스장명: 우리동네 헬스장
  기구: 바벨 스쿼트 렉 (1대, 최대 100kg), 스미스 머신 (1대), 덤벨 (5~40kg)
  특이사항: 오후 6~8시 붐빔

저장 구조:
  문서 1 (요약): "헬스장: 우리동네 헬스장 / 보유 기구 전체: 바벨 스쿼트 렉, 스미스 머신, 덤벨"
  문서 2 (기구별): "기구명: 바벨 스쿼트 렉 / 수량: 1개 / 무게 범위: 최대 100kg"
  문서 3 (기구별): "기구명: 스미스 머신 / 수량: 1개"
  ...
```

각 문서는 `text-embedding-3-small` 모델로 **벡터(숫자 배열)로 변환**되어 ChromaDB에 저장됩니다. 메타데이터로 `user_id`를 함께 저장해 사용자별로 분리합니다.

### 검색 단계 (루틴 생성할 때)

루틴 생성 시 해당 사용자의 `user_id`로 필터링해 기구 정보 전체를 불러옵니다.

```python
# user_id 필터로 해당 사용자 문서만 조회
collection.get(where={"user_id": user_id})
```

반환된 텍스트를 GPT 프롬프트의 `[헬스장 기구]` 섹션에 그대로 주입합니다.

**효과:** GPT는 "이 사용자 헬스장에는 스미스 머신이 있다"는 사실을 알고, 없는 기구(예: 레그프레스)는 루틴에서 자동으로 제외합니다.

## 3. RAG — 공공데이터 기반 추천

### 데이터 출처

- **국민체육공단 체력측정 데이터**: 950건의 실제 측정 기록 (연령대·성별·BMI·체력등급)
- **운동추천 데이터**: 연령대·BMI 등급·성별 조합별 권장 운동 목록

### 저장 방식

각 레코드를 자연어 문장으로 변환 후 ChromaDB에 저장합니다.

```
체력측정 문서 예시:
  "30대 여성, BMI 23.5 (정상), 체력등급 3등급.
   근력: 보통, 유연성: 낮음, 심폐지구력: 보통"

운동추천 문서 예시:
  "30대 여성 BMI 과체중 대상 권장 운동:
   유산소 — 빠르게 걷기 주 3회, 수영 주 2회
   근력 — 스쿼트, 런지, 플랭크"
```

### 검색 단계

사용자의 나이·성별·BMI를 계산한 뒤 유사도 검색을 수행합니다.

```
쿼리 예시: "35세 여성 BMI 24.8 과체중"

→ ChromaDB에서 유사한 체력측정 사례 Top-3 반환
→ 해당 조합에 맞는 운동추천 문서 반환
→ GPT 프롬프트 [유사 체력측정 사례] / [추천 운동 정보] 섹션에 주입
```

**효과:** GPT가 "비슷한 연령·체형의 사람들에게 실제로 권장된 운동"을 근거로 루틴을 구성합니다. 단순 추측이 아닌 데이터 기반 추천이 됩니다.

## 4. GPT 프롬프트 주입 구조

루틴 생성 시 GPT에게 보내는 프롬프트는 여러 섹션을 조합해 구성됩니다.

```

  이름: OOO / 나이: 35세 / 성별: 여성
  키: 165cm / 몸무게: 65kg / BMI: 23.9 (정상)
  체력 수준: intermediate / 목표: 체중감량, 근력향상
  건강 주의사항: 허리 디스크 주의
  부상 태그: 허리

[부상 주의사항]
  허리 부상이 있으므로 데드리프트, 굿모닝 등 허리에 직접 부하가 가는 운동 제외.
  대체 운동: 레그프레스, 케이블 풀다운 권장.

[오늘 볼륨 조정]
  컨디션: 보통 (3/5)
  → 세트 수를 권장 대비 -1세트, 강도 80%로 조정할 것.
  근육통 부위: 어깨 → 어깨 운동 제외.

[오늘 제외할 부위]
  어제 훈련 부위: 가슴, 삼두
  오늘 근육통: 어깨
  → 가슴 / 삼두 / 어깨 운동 제외.

[볼륨 가이드라인]
  목표 체중감량 + 근력향상:
  - 복합 운동 위주 / 세트당 8~12회 / 세트 수 3~4
  - 인터벌 유산소 10~15분 포함 권장

[점진적 향상 분석]
  - 스쿼트: 3회 수행 | 최근 3세트×12회 / 50kg | ⬆ 레벨업 권장 | 무게 53.75kg 시도
  - 플랭크: 2회 수행 | 최근 3세트×60초 | → 현행 유지

[운동 순환 스케줄 — 쿨다운 기반]
  오늘 수행 가능: 하체 (5일 경과), 복근 (3일 경과)
  쿨다운 중 (제외): 가슴 (1일), 삼두 (1일), 어깨 (0일)
  우선 추천: 하체 (가장 오래 쉼)

[헬스장 기구]
  헬스장: 우리동네 헬스장
  보유 기구: 바벨 스쿼트 렉, 스미스 머신, 덤벨 5~40kg, 케이블 머신
  특이사항: 오후 6~8시 붐빔

[유사 체력측정 사례]
  30대 여성 BMI 정상 체력등급 3등급 — 근력 보통, 유연성 낮음

[추천 운동 정보]
```

## 5. 운동 루틴 생성 전체 흐름

```
1. page_recommend() 진입 (Streamlit 페이지)
        ↓
2. 오늘 날짜 키로 컨디션 입력 여부 확인
   → 미입력 시: 슬라이더 + 멀티셀렉트 표시 후 대기
   → 입력 완료: session_state["condition_2026-04-28"] 에 저장
        ↓
3. api_get_today_routine() 호출
   → 오늘 이미 저장된 루틴 있으면 GPT 호출 없이 재사용 (비용 절약)
        ↓
4. 없으면 신규 생성:
   a. api_get_recent_exercises() — 최근 7일 운동 목록 조회
   b. _get_exercise_cooldown_info() — 운동별 쿨다운 계산
   c. _get_recent_body_parts() — 최근 2일 훈련 부위 추출
   d. api_get_rag_context() — 공공데이터 RAG 컨텍스트 요청
   e. retrieve_gym_context() — 헬스장 기구 ChromaDB 검색
   f. _build_routine_prompt() — 모든 정보 조합해 프롬프트 생성
        ↓
5. GPT-4o-mini 호출 → JSON 반환
        ↓
6. api_save_routine() — MySQL routines 테이블에 저장
        ↓
7. 화면 표시: 도넛 차트(부위별 비율) + 바 차트(세트×무게) + 운동 카드
```

## 6. 쿨다운 & Split 자동 감지

### Split 감지 (최근 2일 훈련 부위)

`workout_logs`에서 최근 2일 운동 이름을 가져와 **정규식(regex)으로 신체 부위를 추출**합니다.

```python
# 운동명 → 부위 매핑 예시
"스쿼트" → "하체"
"벤치프레스" → "가슴"
"풀업" → "등"
"오버헤드 프레스" → "어깨"
```

추출한 부위 목록을 프롬프트의 `[오늘 제외할 부위]` 섹션에 넣어 GPT가 해당 부위를 자동 회피합니다.

### 쿨다운 계산

`progression API`에서 운동별 마지막 수행일을 가져와 **경과 일수를 계산**합니다.

```python
쿨다운 기준:
  가슴 / 어깨 / 삼두 / 이두  → 2일 휴식 필요
  하체 / 등                  → 3일 휴식 필요
  복근 / 유산소              → 1일 휴식 필요

경과일 < 기준 → "쿨다운 중" (제외)
경과일 >= 기준 → "수행 가능" (추천)
경과일이 가장 긴 운동 → "오늘 우선 추천"
```

이 결과를 GPT 프롬프트에 주입해, GPT가 쉰 지 오래된 부위를 우선적으로 루틴에 포함시킵니다.

## 7. 점진적 과부하(Progressive Overload) 분석

운동 기록이 쌓이면 **Python 코드가 자동으로 다음 목표를 계산**합니다. GPT가 판단하는 게 아니라 규칙 기반으로 동작합니다.

### 레벨업 판단 기준

아래 3가지를 모두 만족하면 "레벨업 권장" 상태가 됩니다.

```
① 해당 운동 기록이 2회 이상 존재
② 최근 기록의 세트 수 >= 3
③ 평균 반복 수 >= 11회
```

### 다음 목표 계산

```
맨몸 운동 (weight_kg = 0):
  → 횟수 +2회 제안
  또는 더 어려운 변형 동작 제안 (예: 푸시업 → 다이아몬드 푸시업)

기구 운동:
  → 현재 무게 × 1.075 (약 7.5% 증가)
  예: 50kg → 53.75kg

나이/BMI/성별/체력수준 보정:
  고령(60대+) 또는 고도비만 → 증가폭 절반으로 줄임
  초급자 → 표준 증가, 상급자 → 증가폭 확대
```

계산 결과는 GPT 프롬프트 `[점진적 향상 분석]` 섹션에 주입되고, 화면의 "성장 대시보드"에도 표로 표시됩니다.

## 8. 운동 GIF 가져오는 방법

운동 카드에 표시되는 GIF는 **RapidAPI ExerciseDB**에서 가져오며, **3단계 캐싱 전략**으로 API 호출을 최소화합니다.

### 3단계 조회 순서

```
1단계 — DB 캐시 확인
  exercises 테이블에 gif_url(로컬 경로)이 있으면 바로 반환
  → RapidAPI 호출 없음

2단계 — 로컬 GIF 다운로드
  exercise_id가 DB에 있으면:
  RapidAPI /image?exerciseId={id} 호출
  → GIF 파일을 서버 로컬(/static/gifs/)에 저장
  → DB gif_url 업데이트
  → 이후 요청은 1단계에서 처리

3단계 — 운동명 검색으로 ID 획득
  exercise_id도 없으면:
  RapidAPI /exercises/name/{이름} 검색
  → exercise_id 획득
  → 2단계 수행

  검색 실패 시 단어를 줄여가며 재시도:
  "smith machine squat" 실패
  → "smith squat" 재시도
  → "smith" 재시도
```

### 운동 목록 동기화 (`/db/exercises/sync`)

최초 1회 실행하면 RapidAPI에서 **1500개 운동 목록 전체**를 페이지네이션으로 가져와 `exercises` 테이블에 저장합니다. 이후 GPT가 루틴을 생성할 때 이 목록을 참고해 영문명·카테고리가 정확한 운동을 추천합니다.

## 9. 사용자 인증 구조

### 08_FitStep_API — API Key 방식

```
클라이언트 요청 헤더: X-API-Key: <key>
서버: 환경변수 RAG_API_KEY와 비교
일치 → 통과 / 불일치 → 401 반환
RAG_API_KEY 미설정 시 → 인증 비활성화 (개발 편의)
```

비밀번호는 **SHA-256 해시**로 저장합니다. 평문은 절대 DB에 저장되지 않습니다.

```
사용자 입력: "mypassword123"
저장값: hashlib.sha256("mypassword123".encode()).hexdigest()
       → "ef92b779..." (64자리 고정 해시)
로그인 시: 입력 비밀번호를 동일하게 해시 → DB 값과 비교
```

### 09/10 — JWT 방식

에이전트 서버는 사용자별 세션이 필요하므로 JWT(JSON Web Token)를 사용합니다.

```
로그인 성공
  → 서버가 JWT 토큰 발급 (사용자 ID + 만료시간 포함, JWT_SECRET_KEY로 서명)
  → 클라이언트가 토큰 저장

이후 요청
  → Authorization: Bearer <token> 헤더 포함
  → 서버가 서명 검증 + 만료 확인 → 사용자 ID 추출
```

## 10. 멀티에이전트 파이프라인 상세

10_MultiAgent는 단일 GPT 호출 대신 **역할이 다른 4개 에이전트가 순서대로 협력**합니다.

### Step 1 — 전처리 (GPT 없음)

```
match_exercises(equipment):
  사용자 헬스장 기구 목록 → Python 딕셔너리로 수행 가능한 운동 매핑
  예: "바벨 스쿼트 렉" → ["바벨 스쿼트", "바벨 런지", "바벨 힙스러스트"]

search_rag(profile):
  ChromaDB에서 유사 체력측정 사례 + 운동추천 컨텍스트 검색

search_web(query):
  부상·질환 키워드가 있을 때만 Tavily 웹 검색 실행
  없으면 건너뜀
```

### Step 2 — Orchestrator (GPT 1회)

```
입력: 사용자 메시지 + 프로필
출력: {"strength": true, "cardio": true, "rehab": false}

어떤 전문가가 필요한지 GPT가 판단합니다.
부상 관련 언급 없으면 rehab=false로 Rehab Agent 건너뜀.
```

### Step 3 — 전문가 에이전트 (GPT 각 1회)

```
Strength Agent:
  → 근력 세션 설계 (근육 분할, 세트·횟수·무게, 점진적 과부하)

Cardio Agent:
  → 유산소 세션 설계 + 심박수 존 자동 계산
  → 최대 심박수 = 220 - 나이, 목표 존 = 60~80%

Rehab Agent:
  → 금지 운동 목록 + 안전 가이드라인 생성
  → 웹 검색 결과(부상 관련 정보) 참고
```

### Step 4 — Integration Agent (GPT 1회)

```
3개 전문가 결과를 받아 최종 주간 커리큘럼 JSON으로 통합.
Rehab Agent가 금지한 운동은 Strength 결과에서 자동 제거.
요일별·시간별 충돌 방지.
```

### Step 5 — Python 검증 (GPT 없음)

```
검증 항목:
  ① 총 운동 일수가 사용자가 설정한 주당 횟수와 일치하는가
  ② 동일 근육 그룹이 연속 이틀 이상 배치되지 않았는가
  ③ 세션당 총 운동 시간이 합리적인 범위인가 (15~120분)

실패 시: 오류 메시지를 Integration Agent에 전달 → 재생성 (최대 2회)
통과 시: curricula 테이블에 저장
```

## 11. Hybrid Search 상세

09/10에서 공공데이터를 검색할 때는 단순 벡터 검색이 아닌 **3단계 복합 검색**을 사용합니다.

### 왜 Hybrid Search인가?

벡터 검색만 사용하면 의미는 비슷하지만 특정 단어(예: "무릎", "고도비만")가 정확히 매칭되지 않는 경우를 놓칠 수 있습니다. BM25(키워드 검색)와 결합하면 보완됩니다.

### 3단계 파이프라인

```
1단계 — Vector Search (ChromaDB)
  쿼리를 text-embedding-3-small으로 벡터화
  → ChromaDB에서 코사인 유사도 Top-10 반환

2단계 — BM25 Search
  kiwipiepy 한국어 형태소 분석기로 쿼리 토크나이징
  → TF-IDF 기반 키워드 매칭 Top-10 반환

3단계 — RRF 융합 (Reciprocal Rank Fusion)
  두 검색 결과의 순위를 수식으로 결합
  → 양쪽 모두에서 높은 순위의 문서에 높은 점수 부여

4단계 — Reranker
  ms-marco-MiniLM-L-12-v2 모델 (flashrank)로 최종 관련도 재평가
  → Top-3 최종 반환
```

### 결과

단순 벡터 검색 대비 **특정 키워드 매칭 정확도**와 **의미 유사도** 두 가지를 동시에 확보합니다.
