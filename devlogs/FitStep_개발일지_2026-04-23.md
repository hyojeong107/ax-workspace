## 1. 점진적 과부하 알고리즘 개선 — 횟수 무한 증가 문제

### 문제

`07_FitStep_Web/app.py`의 `_build_progression_context_from_api` 함수에서
레벨업 기준이 단순히 "세션 2회 이상 + 세트 3개 이상 + 평균 11회 이상"으로 고정되어 있었음.

```python
# 기존 코드 (문제)
ready = sessions >= 2 and last["sets_done"] >= 3 and avg_reps >= 11.0
status = "⬆ 레벨업 권장" if ready else "→ 현행 유지"
# 맨몸 운동 레벨업 시
suggestion = f"횟수를 {int(avg_reps)+2}회로 늘려보세요"
```

기준을 충족할 때마다 횟수를 +2씩 올리는 구조라 장기적으로 100회 이상으로 증가하는 비현실적인 문제가 있었음.
또한 사용자의 나이, 성별, BMI, 체력 수준을 전혀 반영하지 않아 개인화가 부재했음.

### 원인 분석

- 레벨업 기준이 단일 고정값(11회)으로, 개인 신체 조건 무관
- 맨몸 운동의 횟수 상한이 없어 무한 증가 가능
- 기구 운동도 무게 증가율이 항상 7.5%로 고정 (고령자/고BMI 사용자에게 부적합)

### 해결

사용자 신체 정보(나이, 성별, BMI, 피트니스 레벨)를 반영하는 `_calc_progression_suggestion` 함수를 분리 구현.

**횟수 상한(reps_cap) 개인화:**

| 조건 | 상한 |
|------|------|
| 여성 또는 50세 이상 또는 BMI 30+ | 15회 |
| 초보자(beginner) | 15회 |
| 중급자(intermediate) | 18회 |
| 고급자(advanced) | 20회 |

**레벨업 기준:** 상한의 80% 도달 시 (예: 초보자는 12회 평균)

**무게 증가율 개인화:**

| 조건 | 증가율 |
|------|--------|
| 50세 이상 또는 BMI 28+ | 5% |
| advanced | 5% |
| 그 외 | 7.5% |

**맨몸 운동 상한 도달 시:** "더 어려운 변형 동작으로 교체 필요"로 표시, 실제 변형 동작 추천은 GPT에 위임

```python

    # 상한 80% 도달 시 레벨업 판단
    # 기구: 개인화 증가율 / 맨몸 상한 미달: +2 / 상한 도달: GPT 위임
```

### 함께 수정된 부분

- `recommend_routine`에서 `user` dict를 `_build_progression_context_from_api`에 전달
- 대시보드 "성장 현황 테이블"도 동일 로직 사용 (기존 중복 코드 제거)

## 2. 변형 동작 추천 방식 — 하드코딩 vs GPT 위임

### 문제 및 시도

맨몸 운동 상한 도달 시 "변형 동작으로 전환하세요"라는 추상적인 메시지만 출력되는 문제를 개선하려 함.
1차 시도로 운동명-변형 동작 매핑 테이블을 하드코딩으로 구현:

```python
# 1차 시도 — 폐기
_BODYWEIGHT_PROGRESSIONS: dict[str, list[str]] = {
    "스쿼트": ["고블릿 스쿼트", "불가리안 스플릿 스쿼트", ...],
    "푸시업": ["다이아몬드 푸시업", "아처 푸시업", ...],
    ...
}
```

### 문제점

하드코딩 방식은 운동 목록이 수백 개인 ExerciseDB 기반 앱에서 유지보수가 불가능하고,
사용자 체력/헬스장 환경을 반영하지 못함. GPT가 이미 컨텍스트를 갖고 있으므로
GPT에 위임하는 것이 더 적합한 구조.

### 해결

하드코딩 테이블(`_BODYWEIGHT_PROGRESSIONS`, `_get_next_variation`) 전부 제거.
대신 GPT 프롬프트에 명시적으로 지시:

```
[점진적 과부하 & 순환 전략 — 반드시 준수]
- '⬆ 레벨업 권장' 운동은 제안된 변형 동작 또는 증가된 무게로 오늘 루틴에 반영하세요.
- 변형 동작 제안 시 반드시 실제 운동명을 명시하세요.
  (예: 스쿼트→불가리안 스플릿 스쿼트, 푸시업→다이아몬드 푸시업, 플랭크→RKC 플랭크)
- '→ 현행 유지' 운동도 3~4회 이상 반복됐으면 루틴에 다시 포함시켜 근육이 익숙해지게 하세요.
- 영원히 새 동작만 추가하지 말고, 이전에 했던 운동을 주기적으로 재등장시켜 강도·무게를 높이는 방식으로 순환하세요.
```

### 핵심 설계 원칙

**순환 구조**: 점진적 향상 후 동일 운동을 더 높은 강도로 반복하는 것이 실제 운동 프로그래밍의 기본.
영원히 새 동작만 추가하는 건 현실적으로 불가능하므로, GPT 프롬프트에서 "주기적 재등장 + 강도 순환"을 명시적으로 지시함.

## 3. RAG 성능 개선 검토 — 현재 구조 분석

### 문제 발견

`retrieval.py`의 `retrieve_context`가 실제 벡터 유사도 검색(`collection.query()`)을 전혀 사용하지 않고
`collection.get(where={"user_id": ...})`으로 전체 문서를 그냥 가져오는 구조였음.

```python
# 현재 — 검색이 아니라 전체 가져오기
results = collection.get(where={"user_id": {"$eq": user_id}})
```

즉 RAG라기보다 "DB 전체 조회 후 GPT 프롬프트에 붙이기" 수준.

### RAG 고도화 기법별 이 프로젝트 적합성 검토

| 기법 | 효과 | 이유 |
|------|------|------|
| Indexing 고도화 | 낮음 | GPT가 기구명만으로 충분히 판단 가능 |
| Contextual Retrieval | 낮음 | 기구 문서 수가 적어 오버엔지니어링 |
| Hybrid Search | 보통 | query() 전환 선행 필요, 기구명 정확 매칭에는 유효 |
| Reranking | 낮음 | 전체 get() 구조라 재정렬 대상 자체가 없음 |

### 결론

고도화 기법 적용보다 **개인맞춤화 품질 향상**이 이 서비스의 핵심 가치에 더 부합함.
RAG 개편 방향을 "기술 스택 고도화"가 아닌 "사용자 상태를 구조적으로 반영하는 컨텍스트 생성"으로 전환.

## 4. 개인맞춤화 + 점진적 향상 — 전체 문제 분석 및 개선 계획

### 현재 구조의 근본 문제

```
사용자 정보 → 텍스트로 GPT 프롬프트에 붙임 → GPT가 알아서 해석
```

- `health_notes = "허리 좀 아픔"` 자유 텍스트 → GPT 해석 불일관
- 가입 시 입력한 정보(fitness_level, goal, health_notes)가 영구 고정
  → 부상이 나아도, 운동 실력이 늘어도 반영 안 됨
- 오늘 컨디션(피로도, 근육통 부위) 반영 없음
- 어제 어떤 부위 운동했는지 고려 안 함 (운동 split 개념 없음)
- sets/reps를 GPT가 임의로 결정 (목표별 볼륨 기준 없음)
- fitness_level이 가입 시 선택값으로 영원히 고정

### 개선 항목 및 우선순위

| 항목 | 내용 | DB 변경 | 우선순위 |
|------|------|---------|---------|
| 부상 태그 구조화 | health_notes 자유 텍스트 → 선택형 체크박스 + injury_tags 컬럼 추가 | O | ★★★ |
| 프로필 수정 기능 | 가입 후 모든 정보 수정 가능한 페이지 추가 | O (PATCH API) | ★★★ |
| 오늘의 컨디션 입력 | 루틴 추천 전 컨디션 슬라이더 + 근육통 부위 선택 | X | ★★★ |
| 운동 split 자동 감지 | 최근 2일 운동 부위 분석 → 오늘 제외할 부위 프롬프트 반영 | X | ★★★ |
| 목표별 볼륨/강도 기준 | 목표별 sets×reps 기준을 프롬프트에 명시 | X | ★★★ |
| fitness_level 자동 승급 | 누적 기록 기반 레벨업 제안 배너 | X | ★★☆ |

### 구현 설계

**1. injury_tags 컬럼 추가 (users 테이블)**
```sql
ALTER TABLE users ADD COLUMN injury_tags VARCHAR(200) DEFAULT '';
```
선택 옵션: 허리 통증/부상, 무릎 통증/부상, 어깨 통증/부상, 손목 통증/부상, 없음
→ 콤마 구분 문자열로 저장. 회복 시 프로필 수정에서 직접 제거 가능.

**2. PATCH /db/users/{user_id}/profile API 추가**
업데이트 가능 필드: weight_kg, height_cm, age, fitness_level, goal, health_notes, injury_tags

**3. 오늘의 컨디션 — session_state 저장 방식**
```python
st.session_state[f"condition_{date.today()}"] = {
    "condition": "보통",  # 매우 피곤/피곤/보통/좋음/최상
    "soreness": ["가슴", "어깨"]  # 근육통 부위
}
```
날짜 키로 저장 → 하루 지나면 자동 만료.

**4. 운동 split 감지 — 키워드 매핑**
```python
BODY_PART_MAP = {
    "chest|bench|fly|플라이|벤치|가슴": "가슴",
    "back|row|pulldown|deadlift|등|로우|풀다운|데드": "등",
    "shoulder|press|overhead|어깨|숄더": "어깨",
    "squat|leg|lunge|스쿼트|레그|런지": "하체",
    "bicep|tricep|curl|팔|이두|삼두": "팔",
    "abs|plank|crunch|복근|플랭크": "복근",
}
```
최근 2일 workout_logs 기반 → 오늘 제외 부위 프롬프트 섹션 추가.

**5. 목표별 볼륨 기준 (프롬프트 추가)**

| 목표 | 세트×반복 | 강도 |
|------|----------|------|
| 근육증가 | 3-5세트 × 6-12회 | 고중량 (1RM 75-85%) |
| 체중감량 | 3세트 × 12-15회 | 중중량 + 유산소 필수 |
| 체력향상 | 3-4세트 × 10-15회 | 중중량 |
| 재활/부상회복 | 2-3세트 × 15-20회 | 저중량, 통증 없는 범위 |
| 건강유지 | 3세트 × 10-12회 | 중중량 |

**6. fitness_level 자동 승급 조건**
- 최근 4주 완료 루틴 >= 10회
- 레벨업 권장 운동 >= 3개
- 현재 레벨이 advanced 아님
→ 조건 충족 시 page_recommend() 상단에 승급 제안 배너 표시

### 핵심 설계 원칙

부상 정보를 "절대 금지"로 단순 처리하는 게 아니라,
**회복 가능성을 전제로 사용자가 직접 수정할 수 있는 구조**로 설계.
injury_tags를 별도 컬럼으로 분리한 이유도 health_notes(자유 서술)와
구조화된 태그를 분리 관리하기 위함.
