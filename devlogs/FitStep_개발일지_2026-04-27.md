### 1. Streamlit 웹앱 Docker 패키징

**배경**
기존에는 Streamlit을 `streamlit run app.py`로 로컬에서 직접 실행했음.
MySQL + FastAPI는 Docker로 띄우면서 Streamlit만 따로 실행하는 구조가 번거로워, Streamlit도 Docker로 패키징해 `docker compose up` 한 번으로 전체 스택을 실행할 수 있도록 변경.

**변경 파일**

- `07_FitStep_Web/Dockerfile` (신규 생성)
  - chromadb 컴파일 필요로 인해 2단계 빌드(builder/runtime) 적용
  - builder 단계에서 `build-essential`, `gcc` 설치 후 패키지만 runtime에 복사
  - `secrets.toml` 복사 제외 → 환경변수는 docker-compose에서 주입
  - `app.py`가 `../06_FitStep/rag/`를 `sys.path`로 참조하므로 `/06_FitStep/rag/`에 복사
  - build context를 루트(`..`)로 설정해야 두 폴더(06, 07) 모두 접근 가능

```dockerfile
FROM python:3.12-slim AS builder
WORKDIR /build
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential gcc && rm -rf /var/lib/apt/lists/*
COPY 07_FitStep_Web/requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /install /usr/local
COPY 07_FitStep_Web/app.py 07_FitStep_Web/api_client.py 07_FitStep_Web/requirements.txt ./
COPY 07_FitStep_Web/themes/ ./themes/
COPY 06_FitStep/rag/ /06_FitStep/rag/
EXPOSE 8501
CMD ["streamlit", "run", "app.py",
     "--server.port=8501", "--server.address=0.0.0.0", "--server.headless=true"]
```

- `08_FitStep_API/docker-compose.yml` — `streamlit` 서비스 추가

```yaml
streamlit:
  build:
    context: ..
    dockerfile: 07_FitStep_Web/Dockerfile
  container_name: fitstep_streamlit
  ports:
    - "8501:8501"
  environment:
    - OPENAI_API_KEY
    - RAG_API_KEY=${RAG_API_KEY}
    - RAG_API_URL=http://gym-rag:8000   # Docker 내부 네트워크 주소
  env_file:
    - .env
  depends_on:
    gym-rag:
      condition: service_healthy
  restart: unless-stopped
```

**핵심 포인트**
- Docker 내부에서 컨테이너끼리 통신할 때는 `localhost` 대신 서비스 이름 사용 (`http://gym-rag:8000`)
- 로컬 실행(`streamlit run app.py`)은 그대로 유지됨. Dockerfile은 Docker 전용 설정 파일일 뿐

### 3. ModuleNotFoundError: No module named 'rag' 수정

**배경**
`app.py`가 `sys.path`에 `../06_FitStep`을 추가한 뒤 `from rag.gym_rag import ...`로 06폴더의 rag 모듈을 가져옴.
Docker build context가 `07_FitStep_Web/`만이었을 때는 `06_FitStep/rag/`가 컨테이너에 존재하지 않아 import 실패.

**원인**
```
# app.py의 경로 참조 방식
_FITSTEP = os.path.normpath(os.path.join(_STREAMLIT_DIR, "..", "06_FitStep"))
sys.path.insert(0, _FITSTEP)
from rag.gym_rag import ...   # → /06_FitStep/rag/gym_rag.py 를 찾음
```

**수정 내용**
- `docker-compose.yml`의 streamlit build context를 `../07_FitStep_Web` → `..` (루트)로 변경
- `Dockerfile`의 COPY 경로를 루트 기준으로 전체 수정
- `COPY 06_FitStep/rag/ /06_FitStep/rag/` 추가해 컨테이너 안에서도 동일한 경로로 접근 가능하게 설정

```

Database: fitstep
Username: root
Password: .env의 MYSQL_ROOT_PASSWORD 값
```

### 5. GIF 이미지 미표시 버그 수정

**배경**
Docker로 실행 시 운동 루틴 화면에서 GIF 이미지 자리에 "0"이 표시되는 문제 발생.

**원인 분석**
```
1. API가 gif_url로 "/static/gifs/0739.gif" 반환
2. api_client.py가 이를 "http://gym-rag:8000/static/gifs/0739.gif"로 변환
3. st.image("http://gym-rag:8000/...")  호출
4. Streamlit이 이 URL을 그대로 브라우저의 <img> 태그에 전달
5. 브라우저는 "gym-rag"라는 Docker 내부 호스트명을 알 수 없음 → 이미지 로드 실패 → "0" 표시
```

- `gym-rag:8000`은 Docker 내부 네트워크 주소라 Streamlit 컨테이너끼리는 통하지만, **사용자 브라우저**에서는 접근 불가

**수정 내용**
- `07_FitStep_Web/app.py` — 이미지 표시 로직 변경

```python
# 수정 전: URL을 브라우저에 직접 전달 (브라우저가 gym-rag 주소를 못 찾음)
st.image(gif_url, use_container_width=True)

# 수정 후: Streamlit 서버가 직접 다운로드 후 바이트로 전달
try:
    import requests as _req
    img_bytes = _req.get(gif_url, timeout=5).content
    st.image(img_bytes, use_container_width=True)
except Exception:
    st.markdown("🖼 이미지 준비 중")
```

- Streamlit 서버(컨테이너)는 `gym-rag:8000`에 접근 가능하므로 이미지를 서버에서 다운로드한 뒤 바이트로 브라우저에 전달 → 브라우저가 Docker 내부 주소를 알 필요 없음

**검증**
```bash
# Streamlit 컨테이너 안에서 직접 접근 테스트
docker exec fitstep_streamlit python3 -c "
import requests
r = requests.get('http://gym-rag:8000/static/gifs/0739.gif', timeout=5)
print('status:', r.status_code)   # 200
print('bytes:', len(r.content))   # 115136
"
```

### 6. RAG 성능 평가 및 Docker 자동 인덱싱 구조 추가

**배경**
`08_FitStep_API`의 공공데이터 RAG 성능을 평가(`python -m eval.run_eval`)했는데 아래와 같은 결과가 나옴.

| 지표 | 점수 |
|------|------|
| Precision@3 (검색 청크 관련성) | **0.0** |
| Faithfulness (컨텍스트 근거율) | **0.0** |
| Requirement Coverage | 0.95 |
| Rule Check | 1.00 |

**원인 분석**
모든 케이스에서 검색 컨텍스트가 0개였음. ChromaDB의 `fitness_measurement`, `exercise_recommendation` 컬렉션이 비어있고 BM25 pkl 파일도 없었기 때문.

- `init_public_data.py`가 인덱싱 스크립트로 존재하지만 한 번도 실행되지 않은 상태
- Requirement Coverage·Rule Check는 GPT-4o 자체 지식으로 생성한 답변을 평가하므로 정상이었음 (RAG 없이도 좋은 답변 생성)
- `retrieval.py`의 모든 검색 함수가 `except Exception: return ""`로 오류를 숨기는 구조라 문제를 인지하기 어려웠음

**RAG 파이프라인 구조**
```
[쿼리] → retrieve_fitness_context() + retrieve_exercise_recommendation()
          ↓ 각각 ChromaDB 벡터 검색 + BM25 + RRF 융합 + Reranker(ms-marco)
          ↓ contexts = []  ← 데이터 없으면 항상 빈 배열
         GPT-4o 답변 생성 (컨텍스트 없으면 자체 지식으로)
          ↓
        Precision@3=0, Faithfulness=0
```

**원래 Docker 구조의 문제**
기존 Dockerfile은 `app/` 폴더만 복사하고 CMD로 바로 uvicorn을 실행하는 구조였음.
`init_public_data.py`와 `data/` 폴더(CSV, JSON)가 이미지에 포함되지 않아, Docker로 실행하면 ChromaDB가 항상 빈 상태로 시작하고 공공데이터 RAG가 동작하지 않는 구조였음.

**변경 파일**

- `08_FitStep_API/entrypoint.sh` (신규 생성)
  - 컨테이너 시작 시 ChromaDB 데이터 존재 여부를 확인
  - 비어있으면 `init_public_data.py` 자동 실행 후 uvicorn 시작
  - 데이터가 이미 있으면 인덱싱 스킵하고 바로 시작

```sh
#!/bin/sh
# ChromaDB가 비어있으면 공공데이터 인덱싱 후 서버 시작
python - <<'EOF'
...
if needs_index:
    from init_public_data import index_fitness_data, index_exercise_recommendation
    index_fitness_data()
    index_exercise_recommendation()
EOF
exec uvicorn app.main:app --host 0.0.0.0 --port 8000
```

- `08_FitStep_API/Dockerfile` — ENTRYPOINT 방식으로 변경

```dockerfile
# 추가된 COPY
COPY init_public_data.py ./
COPY data/국민연령별추천운동정보.csv ./data/
COPY "data/체력측정 및 운동처방 종합 데이터.json" "./data/..."
COPY entrypoint.sh ./

RUN chmod +x /app/entrypoint.sh
ENTRYPOINT ["/app/entrypoint.sh"]
```

- `08_FitStep_API/docker-compose.yml` — `ANTHROPIC_API_KEY` 환경변수 추가
  - `init_public_data.py`가 Claude Haiku(Contextual Retrieval)를 사용하므로 필요

**동작 방식**
```
첫 번째 docker compose up
  → entrypoint.sh → ChromaDB 비어있음 확인
  → init_public_data.py 자동 실행 (Claude Haiku로 컨텍스트 생성 + ChromaDB + BM25 저장)
  → chroma_data Docker named volume에 영속 저장
  → uvicorn 시작

두 번째 이후 재시작
  → entrypoint.sh → ChromaDB에 데이터 있음 확인
  → 인덱싱 스킵, 바로 uvicorn 시작
```

`chroma_data` Docker named volume이 데이터를 보존하므로 컨테이너를 재시작해도 재인덱싱하지 않음.
단, `docker volume rm chroma_data` 시에는 볼륨이 초기화되어 다음 시작 시 자동으로 재인덱싱됨.

**주의사항**
- 첫 기동 시 Claude Haiku API를 청크 수만큼 호출하므로 시간이 걸림 → HEALTHCHECK `start-period`를 60초로 늘림

## 오늘 새로 알게 된 것

### 로컬 eval과 Docker는 ChromaDB를 공유하지 않는다

**몰랐던 부분**
가상환경에서 `python -m eval.run_eval`을 실행하면 Docker에 올라간 설정 그대로 eval이 되는 줄 알았음.

**실제 동작**
로컬 eval과 Docker 컨테이너는 **완전히 별개의 ChromaDB**를 바라봄.

```
로컬 venv에서 eval 실행
  → app/indexing.py의 CHROMA_PATH
  → 08_FitStep_API/data/chroma_db/  ← 로컬 폴더

Docker 컨테이너 (gym_rag_api)
  → /app/data/chroma_db/  ← chroma_data Docker named volume
```

Docker를 빌드하고 올려도 로컬 `data/chroma_db/`는 그대로 비어있고, 반대로 로컬에서 인덱싱해도 Docker volume엔 영향 없음.

**따라서 eval을 로컬에서 돌리려면 Docker와 무관하게 로컬에서 직접 인덱싱이 필요함**

```bash
cd 08_FitStep_API
source venv/bin/activate
python init_public_data.py   # 로컬 data/chroma_db/ 채움 (한 번만)
python -m eval.run_eval      # 이후 계속 재사용 가능
```

### eval은 Docker 안에서 못 돌린다 (현재 구조)

Dockerfile이 `app/`, `init_public_data.py`, 데이터 파일만 복사하고 `eval/` 폴더는 이미지에 포함하지 않음.
`docker exec`로 컨테이너에 들어가도 eval 폴더 자체가 없음.

eval은 개발·테스트 도구이므로 로컬에서 실행하는 것이 맞고, Docker는 서버 역할만 담당하는 구조가 적절함.

| 상황 | 인덱싱 위치 | eval 가능 |
|---|---|---|
| `python init_public_data.py` (로컬) | `data/chroma_db/` (로컬) | ✅ |
| `docker compose up --build` | `chroma_data` volume (Docker) | ❌ (eval은 로컬 경로 참조) |
| 둘 다 실행 | 각각 독립적으로 채워짐 | ✅ |
