## 오늘 한 작업

### 1. 멀티에이전트 전환 필요성 검토

**배경**
09_SingleAgent는 현재 구조(선형 ReAct 루프)로 충분하지만, 학습 목적으로 멀티에이전트 아키텍처를 적용하기 위해 기능 추가 방향을 먼저 논의함.

**결론: 다음 두 방향이 진짜 멀티에이전트를 정당화함**

| 방향 | 이유 |
|------|------|
| 전문가 팀 시스템 | 근력/유산소/재활 각자 다른 전문 프롬프트 → 하나의 GPT가 전부 알 필요 없어짐 |
| 비GPT 에이전트 | 기구 매칭, 스케줄 최적화, 검증은 Python 코드가 더 신뢰도 높음 |

**핵심 설계 원칙 도출**
> "GPT는 언어를 잘 쓰는 도구. 수학 계산·규칙 적용·필터링은 코드가 담당하고, 자연어 생성만 GPT에 맡긴다."

### 2. 10_MultiAgent 설계

#### 전체 아키텍처

```
사용자 요청
  ↓
Orchestrator (파이프라인 컨트롤러)
  ├── Step 1: match_exercises()     Python 딕셔너리 매핑  (GPT 없음)
  ├── Step 2: search_rag()          ChromaDB 하이브리드 검색 (GPT 없음)
  ├── Step 3: search_web()          Tavily (부상/질환 있을 때만)
  ├── Step 4: _analyze_specialists() GPT 1회 — 어떤 전문가가 필요한지 결정
  ├── Step 5a: run_strength_agent() GPT 1회 — 근력 운동 세션 생성
  ├── Step 5b: run_cardio_agent()   GPT 1회 — 유산소 세션 + 심박수 존 계산
  ├── Step 5c: run_rehab_agent()    GPT 1회 — 금지 운동 + 안전 가이드라인
  ├── Step 6: run_integration_agent() GPT 1회 — 전문가 결과 통합
  └── Step 7: validate_curriculum() Python 규칙 검증 (GPT 없음) → 실패 시 Step 6 재시도 (최대 2회)
```

**싱글 vs 멀티 GPT 호출 비교**

| | 09_SingleAgent | 10_MultiAgent |
|---|---|---|
| GPT 호출 방식 | ReAct 루프 (최대 10회, 비결정적) | 목적별 최대 5회 (결정적) |
| 기구 필터링 | GPT가 "알아서" | Python 코드가 보장 |
| 검증 | Python 1회 | Python 1회 + 재통합 루프 |
| 검증 실패 처리 | GPT가 루프 안에서 판단 | 오케스트레이터가 명시적 재시도 |

#### 주요 구현 결정 사항

**① 프롬프트 외부 파일 분리**

09_SingleAgent는 `prompts/agent_system.txt` 1개로 전부 관리했으나, 10에서는 에이전트별로 분리.
컨테이너 재빌드 없이 프롬프트 수정 가능하도록 docker-compose에 볼륨 마운트 추가:
```yaml
volumes:
  - ./prompts:/app/prompts:ro
```

**② exercise_matcher.py — 핵심 로직**

기구 → 운동 매핑을 Python 딕셔너리로 관리. GPT에 의존하지 않으므로 "바벨 없는 사용자에게 바벨 스쿼트 추천" 같은 할루시네이션이 코드 레벨에서 원천 차단됨.

```python
EQUIPMENT_TO_EXERCISES = {
    "바벨":    ["바벨 스쿼트", "바벨 데드리프트", ...],
    "트레드밀": ["걷기", "조깅", "경사 걷기", ...],
    # ...
}
# 기구 없으면 llm_fallback 모드 → 맨몸 운동 위주 생성
```

**③ validator.py — 검증 항목 명세 추가**

09의 `validate_curriculum()`을 그대로 쓰되, 항목별 `check_summary` 딕셔너리를 추가해 UI에서 어떤 항목이 왜 실패했는지 정확히 표시할 수 있도록 개선.

```python
return {
    "is_valid": len(errors) == 0,
    "errors": errors,
    "warnings": warnings,
    "check_summary": {  # 신규
        "day_count_min": True,
        "session_duration": False,  # ← 이 항목이 실패
        "muscle_group_rule": True,
        ...
    }
}
```

**④ curricula 테이블 신규 추가 (DB 스키마 확장)**

기존 `routines` 테이블은 날짜별 오늘의 운동 저장용. 멀티에이전트가 생성한 주간 커리큘럼은 별도 `curricula` 테이블에 저장.

```sql
CREATE TABLE IF NOT EXISTS curricula (
    id               INT AUTO_INCREMENT PRIMARY KEY,
    user_id          INT NOT NULL,
    label            VARCHAR(200),
    curriculum_json  LONGTEXT NOT NULL,
    specialists_used VARCHAR(100),    -- "strength,cardio,rehab"
    total_days       INT DEFAULT 0,
    is_valid         TINYINT(1) DEFAULT 1,
    validation_json  TEXT,
    created_at       DATETIME DEFAULT NOW(),
    FOREIGN KEY (user_id) REFERENCES users(id)
)
```

기존 테이블(`routines`, `workout_logs`, `users`)은 변경 없음 — 스키마 일관성 유지.

**⑤ eval/ 고도화 — 코드 검증 vs LLM 판단 분리**

09 기반 eval은 단일 스크립트에서 모든 평가를 처리. 10에서는 명확히 분리:

| 파일 | 역할 | GPT 호출 |
|------|------|---------|
| `code_checks.py` | Precision@K, 요구사항 커버리지, 규칙 통과율 | 없음 |
| `llm_judge.py` | Faithfulness, 멀티에이전트 Coherence | 있음 |

멀티에이전트 전용 평가 항목 추가: `judge_multi_agent_coherence()` — 전문가 결과와 최종 커리큘럼의 일관성 검사 (금지 운동이 포함됐는지 등).

**⑥ Streamlit UI 개선 사항**

| 추가 UI | 내용 |
|---------|------|
| 전문가 태그 | 💪 근력 / 🏃 유산소 / 🩺 재활 컬러 태그 표시 |
| 파이프라인 로그 | 각 단계 처리 내역 — 실패는 빨간색, 완료는 초록색으로 구분 |

### 4. 커리큘럼 → 오늘의 운동 연결 미완성 (다음 세션 작업 예정)

**현재 문제**
10_MultiAgent는 주간 커리큘럼을 `curricula` 테이블에 저장하지만, 이를 날짜별로 분해해서 `routines` 테이블에 연결하는 부분이 빠져있음.

**완성해야 할 흐름**

```
① 주간 커리큘럼 생성 (10_MultiAgent /chat)
② "활성화" 버튼 클릭
③ weekly_plan을 날짜별로 분해 → routines 테이블에 INSERT
   예: 오늘(월)=가슴/삼두, 내일(화)=유산소, ...
④ GET /today → 오늘 날짜로 routines 조회
⑤ Streamlit 요일 탭 UI
   [월] [화] [수] [목] [금] [토] [일]
   → 오늘 탭(수요일)에 자동 포커스
   → 오늘의 운동 루틴 상세 표시
⑥ 운동 완료 체크 → workout_logs 기록
```

**다음 세션 작업 항목**
- `POST /curricula/{id}/activate` 엔드포인트 구현
- `GET /today` 엔드포인트 구현 (날짜 조회)
- Streamlit 요일 탭 UI 구현
- 완료 체크 → workout_logs 연동

## 오늘 새로 이해한 것

### 멀티에이전트 ≠ GPT 여러 번 부르기

흔히 "에이전트 = LLM 호출"로 착각하지만, 실제로는 **독립적인 책임과 도구를 가진 처리 단위**가 에이전트임.

```
GPT에만 의존하는 구조 (안티패턴):
  Agent1(GPT) → Agent2(GPT) → Agent3(GPT)
  = 그냥 GPT 3번 부르는 것

올바른 멀티에이전트:
  Matcher(Python) → RAG(ChromaDB) → Specialist(GPT) → Validator(Python)
  = 각 단계가 최적의 도구를 사용
```

### 주간 커리큘럼 vs 오늘의 운동

현재 전체 FitStep 아키텍처에서 두 개념이 분리되어 있음:

| 개념 | 담당 | 저장소 |
|------|------|-------|
| 오늘의 운동 추천 | 08/07 (날짜 기반, 매일 생성) | `routines` 테이블 |
| 주간 커리큘럼 | 09/10 (에이전트 기반, 한 번 설계) | `curricula` 테이블 |

이 둘을 연결하는 "활성화 → 날짜 분해" 레이어가 다음 작업 핵심.
