## 1. 유산소 운동 UI 개선 — sets/reps → duration/speed/incline

### 문제
기존 유산소 운동도 근력 운동과 동일하게 세트 수 × 횟수 × 무게 형식으로 표시되고 있었음.
유산소 운동의 특성과 맞지 않는 UI였음.

### 개선 아이디어
유산소 운동은 **시간(분) / 속도(km/h) / 경사도(%)** 형식으로 표시하고,
동일 칼로리를 소모할 수 있는 대체 유산소 운동 목록을 드롭다운(selectbox)으로 선택할 수 있도록 변경.

### 구현 방법
- GPT 프롬프트에 유산소 전용 출력 규칙 추가:
  - `duration_min`, `speed_kmh`, `incline_pct` 필드 사용
  - MET × 체중(kg) × 시간(h) 공식으로 칼로리 계산
  - 동일 칼로리 소모 가능한 대체 운동 8개 이상 `cardio_alternatives` 배열에 포함
  - 런닝머신, 등산, 수영, 자전거, 줄넘기 등 12개 종목 필수 포함
- Streamlit UI에서 `category == "유산소"`인 운동은 `st.selectbox`로 렌더링

### 트러블슈팅
- **문제**: 처음엔 대체 운동이 4개밖에 안 나옴
- **원인**: GPT 프롬프트 예시에 3개짜리 샘플이 있어서 GPT가 그걸 따라 3~4개만 생성
- **해결**: 프롬프트 예시를 11개 전체로 교체 + "반드시 8개 이상" 명시

## 2. 루틴 캐시 문제 — 같은 루틴만 계속 나오는 현상

### 문제
"새 루틴 받기" 버튼을 눌러도, 심지어 로그아웃하거나 새 탭을 열어도 항상 동일한 루틴이 표시됨.

### 원인 분석
- `recommend_routine()` 함수가 오늘 날짜 루틴이 DB에 있으면 GPT 호출 없이 그대로 반환
- `st.session_state.today_result`에 캐시된 루틴이 남아있어 페이지 재렌더링 시 재사용
- DB에 저장된 루틴이 이전 버그(gym_context 꼬임)로 생성된 잘못된 루틴이었음

### 해결
1. `force_new` 파라미터 추가: `recommend_routine(user, force_new=False)`
2. "새 루틴 받기" 클릭 시:
   - DB에서 오늘 루틴 삭제 (`DELETE /db/routines/today/{user_id}`)
   - `st.session_state.today_result = None`
   - `force_new_routine = True` 플래그 설정 후 `st.rerun()`
3. FastAPI에 `DELETE /db/routines/today/{user_id}` 엔드포인트 추가
4. `api_client.py`에 `api_delete_today_routine()` 함수 추가

## 3. ChromaDB 쿼리 버그 — 다른 유저의 헬스장 기구가 다른 유저에게 적용되는 현상

### 문제
user_id=4가 덤벨 + 케틀벨을 헬스장 기구로 등록했는데,
다른 유저(user_id=3, 5, 6)로 로그인해서 루틴 추천을 받아도
덤벨 + 케틀벨 + 맨몸운동만 추천되는 현상 발생.

### 원인 분석
ChromaDB의 신버전에서 `where={"user_id": user_id}` 형식이 동작하지 않음.
반드시 `where={"user_id": {"$eq": user_id}}` 형식을 사용해야 함.
`$eq` 연산자 누락으로 필터가 제대로 작동하지 않아 전체 데이터가 반환되거나
예상치 못한 동작이 발생.

### 수정 파일 및 위치
| 파일 | 수정 내용 |
|------|-----------|
| `08_FitStep_API/app/indexing.py` | 기존 데이터 삭제 시 `$eq` 누락 수정 |
| `08_FitStep_API/app/retrieval.py` | `retrieve_context`, `gym_exists` 함수 `$eq` 수정 |
| `06_FitStep/rag/gym_rag.py` | `retrieve_gym_context`, `has_gym_data` 등 3곳 `$eq` 수정 |

### 핵심 코드 변경
```python
# Before (버그)
collection.get(where={"user_id": user_id})

# After (수정)
collection.get(where={"user_id": {"$eq": user_id}})
```

## 4. load_dotenv 경로 문제 — RAG API URL 환경변수 미로드

### 문제
`gym_rag.py`에서 `RAG_API_URL` 환경변수를 읽지 못해
API 모드가 아닌 로컬 ChromaDB 모드로 폴백되는 현상.

### 원인 분석
`app.py` 상단에서 `os.chdir()`로 작업 디렉토리를 `06_FitStep`으로 변경함.
`gym_rag.py`의 `load_dotenv()`는 모듈 임포트 시 단 한 번 실행되는데,
이때 상대경로로 `.env`를 찾아서 잘못된 위치의 `.env`를 로드하거나 로드 실패.

### 해결
`gym_rag.py`의 `load_dotenv()`를 절대경로 기반으로 수정:
```python
# Before
load_dotenv()

# After
load_dotenv(os.path.join(os.path.dirname(os.path.abspath(__file__)), "..", ".env"))
```
또한 `app.py` 최상단에서 두 `.env` 파일을 명시적으로 선로드:
```python
load_dotenv(os.path.join(BASE_DIR, ".env"), override=True)         # 07_FitStep_Web/.env
load_dotenv(os.path.join(GYM_DIR, ".env"), override=False)         # 06_FitStep/.env
```

## 5. 헬스장 미등록 유저 루틴 다양성 문제

### 문제
헬스장 기구를 등록하지 않은 유저도 케틀벨, 덤벨, 맨몸운동 위주로만 추천받는 현상.

### 원인
- `gym_context`가 비어있을 때 프롬프트에 아무 기구 정보도 없었음
- GPT가 기구 제한 없는 상황에서 "안전한" 기본 운동만 선택하는 경향

### 해결
헬스장 미등록 시 일반 헬스장 기구 목록을 프롬프트에 명시:
```python
gym_section = "\n[헬스장 환경]\n일반 헬스장 기준으로 바벨, 덤벨, 케이블 머신, 스미스 머신, " \
              "레그프레스, 랫풀다운, 체스트 플라이 머신, 시티드 로우, 레그 컬, 레그 익스텐션 등 " \
              "모든 기구를 자유롭게 사용할 수 있습니다. 매 루틴마다 다양한 기구를 활용해 " \
              "변화 있는 운동을 구성해주세요.\n"
```
